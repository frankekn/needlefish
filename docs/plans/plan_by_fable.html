§0 給接手者:本文件使用規則

你(接手的 LLM 或人)在動任何東西之前,先把這十條讀完。這十條的存在理由:過去六個月裡,這個專案每一次品質躍升都來自紀律,每一次差點翻車都來自即興。

  2. 動手前先驗證引用。本文件所有 file:line 以 commit df60aac 為準。開工第一步:git log --oneline -3 確認你在哪;若 main 已前進,先 grep 確認引用的符號還在原位,對不上就停,不要憑記憶腦補。
  3. 一次只做一張工單。做完該工單「驗收點」表的全部項目(跑檢驗指令、比對通過標準、存證輸出),才准碰下一張。並行做兩張 = 兩張都做壞。

  6. 你認為本文件寫錯了 → 停,報告,等 Frank 裁決。不要自行修改本文件之後照著改過的版本做。

§1 這個專案是什麼(一頁)

Needlefish 是一個嚴格的 PR 審查 agent:TypeScript CLI,把 local diff 或 GitHub PR 餵給 read-only 的模型 runner(預設 codex gpt-5.5),用 critic pass 刪掉弱 finding,最後由確定性程式碼(不是模型)決定 verdict。三個產品支柱,任何改動不得動搖:

  • Evidence-or-silence:每個 blocking finding 必須附 checked[] 結構化證據;給不出證據就不准報。寧可零 finding,不可編故事。
  • Read-only:模型 runner 在沙箱裡只能讀(rg/git show),永不寫入目標 repo、永不碰 GitHub 憑證。
  • 決定論 verdict:src/core/verdict.ts 的 deriveVerdict 只看驗證過的 findings/residual risks——P0–P2 存在 → changes_requested;blocking residual → needs_human;否則 pass。模型散文永遠不能決定 pass/fail。
審查管線(兩條路,由 diff 大小自動選): 小徑(預設):bundle ──> review(1 次調用)──> critic(1 次調用)──> normalize ──> deriveVerdict ──> render/post 大徑(大 diff):bundle ──> map(拆 hotspot)──> deep × N(併發 3)──> critic ──> normalize ──> deriveVerdict ──> render/post 三種介面: local CLI needlefish --repo <path> → stdout Markdown + ~/.cache/needlefish/<repo>/last-review.json local PR needlefish pr <number> → 同上,免切 branch GitHub mode action.yml / review.yml → COMMENT review + 行內留言 + Needlefish check-run(sticky:同 PR 更新同一則)

§3 憲法:十二條紅線

每一條都是用真實的失敗或量測換來的。格式:條文 → 為什麼 → 怎麼驗證自己沒違反。

第 1 條 — Eval gate 是唯一出貨通道。任何 prompts/ 或審查管線(src/core/、src/shared/normalize.ts、runner 行為)的改動,必須:動手前寫死通過判準 → 全量 fixture @ 預設 effort 跑 gate → 不過就 revert,數據照樣記進 eval/RESULTS.md。
為什麼:gating sweep 功能就是被自己的 gate 否決後 revert 的(102b270)——這個先例證明判準寫死能擋住「感覺有用」的功能。
驗證:你的 PR/commit 訊息裡有 gate 數字,且判準是在跑 gate 之前寫下的(工單裡已寫死的就用工單的)。
第 2 條 — Prompt 禁用 fixture 詞彙。prompt 措辭不得包含任何 eval fixture 的專有名詞、變數名、語言特定慣用語。類別定義必須通用。
為什麼:否則 eval 分數上升是背答案,不是變聰明,真實 PR 上立即現形。
驗證:§0 第 7 條的 grep 自檢。
第 3 條 — Holdout 封存紀律。標了 holdout: true 的 fixture,prompt 措辭迭代期間永不跑(--holdout exclude);只在最終 gate 跑(--holdout include,這也是預設)。每輪 prompt 改動至少新增一個 holdout fixture。
為什麼:34 個 fixture 全參與調參 = 遲早全體過擬合;holdout 是唯一的體外對照組。
驗證:迭代期間的報告 JSON 裡 holdout 欄位是 exclude。
第 4 條 — Verdict 決定論。deriveVerdict 之外的任何地方(尤其是模型輸出的散文)不得影響 pass/fail。改 verdict 邏輯 = 改產品契約,需要 Frank 簽核。
驗證:src/core/verdict.ts 仍然只有那 ~14 行純函式。
第 5 條 — Runner 永遠 read-only。不得給模型 runner 任何能變異目標 repo 或觸碰 GitHub 憑證的權限。runCodex 剝除 token env、沙箱唯讀,這一層不准鬆。
驗證:src/shared/codex.ts 與 runner-sandbox.ts 的既有測試全綠;新增 runner 時比照辦理。
第 6 條 — 單抽不下結論。全量單抽在 medium 下天然閃爍 1–2 個 fixture。任何「回歸了」或「變好了」的結論,必須對 divergent fixtures confirm × 3 draws 之後才成立。
驗證:你引用的每個結論數字旁邊都有 draws 數。
第 7 條 — 不為單一目標 repo 客製。不加目標 repo 的名詞、欄位、fixture、bug 特定 prompt 補丁。Needlefish 是通用工具。
第 8 條 — Commit 不留 AI attribution。無 Co-Authored-By、無 Generated with。違者 reject。
第 9 條 — --fix 與 multi-repo config 不做,除非 Frank 明示立項。--fix 的立項前提是 W2 suggestion 的實際使用率數據。
第 10 條 — 外部留言正文永不進 prompt。@needlefish 指令是嚴格白名單前綴解析,只有 sanitize 過的 finding 匹配鍵能到模型面前。外部留言者是不可信輸入(prompt injection 防線)。
驗證:commands.yml 的 parse job 仍是 case-pattern 精確匹配。
第 11 條 — Miss museum。真實 PR 上每個確認的 miss(人抓到、bot 沒抓到),一週內化成 generic fixture(無目標 repo 詞彙)進 eval。這是 recall 持續上升的唯一原料來源。
第 12 條 — 結構優先於措辭。模型明明有證據卻漏掉 bug 時,優先改 process/輸出結構(如 TRIGGER 證據契約),其次才是 prompt 散文。散文補丁的半衰期很短。
為什麼:P9 六輪迭代的每一輪都證明:有效的修法都是結構(critic 例外、證據欄位),失效的都是形容詞。

4.4 標準 eval gate SOP(prompt/管線改動一律走這裡)

  1. 寫死判準(動手改碼之前,寫進工單或 commit draft):最低 recall、fp 上限、invalidJson 上限、目標 fixture 的達標數、meanDur 漲幅上限。
  2. 若本輪會改 prompt 措辭:先寫一個新 holdout fixture 並封存(spec 標 holdout: true)。
  3. 迭代期:--holdout exclude --fixtures '<相關 fixture regex>' --draws 1 快速反饋,只跑相關子集。
  4. 措辭定稿:全量 gate --holdout include --draws 1 --report eval/reports/<工單>-gate.json。
  5. 對 divergent fixtures(單抽與基線不一致的)confirm:--fixtures '<id>' --draws 3。
  6. 判準全過 → commit(數據入訊息)+ 把報告留在 eval/reports/,重要 gate 複製到 eval/results/ + 更新 eval/RESULTS.md(格式模仿既有段落:表格 + Reports 行)。
  7. 判準不過 → 最多再迭代一輪;兩輪不過就 revert,RESULTS.md 記失敗數據與嘗試過的措辭(見 PB1)。
陷阱:改任何 prompts/*.md 都會改變 promptHash,跨 hash 的數字不可比。A/B 必須在同一 hash 內設臂,或重跑基線臂。

4.5 發版 SOP

  1. CHANGELOG.md:把 Unreleased 內容 cut 成 ## X.Y.Z — YYYY-MM-DD。
  2. package.json 的 version 改成 X.Y.Z。
  3. commit chore(release): X.Y.Z → push(push 需 Frank 同意,§7)。
  4. git tag vX.Y.Z && git push origin vX.Y.Z → release.yml 自動:跑 pnpm check && pnpm test → 驗證過才滾浮動 major tag(v0 或 v1)→ 建 GitHub Release(generate-notes)。
  5. 驗收:gh release view vX.Y.Z 存在;git ls-remote origin refs/tags/v0 指到新 SHA。

回滾(發錯版時,一分鐘):git tag -f v0 <上一個好版的 SHA> && git push -f origin v0。不可變 tag(vX.Y.Z)永不改。

4.6 Weekly eval 看守

.github/workflows/weekly-eval.yml:每週日 18:00 UTC 在 self-hosted runner 跑全量 × 3 draws @ medium,報告 commit 進 eval/results/weekly/。兩種 issue 會自動開:

  • recall 掉 >5pp → 照 PB4 處理(先 confirm 再診斷,不要直接改 prompt)。
  • run failed → 通常是 runner 機器或 codex CLI 問題,照 PB5。

手動觸發:GitHub Actions 頁面 workflow_dispatch,或 gh workflow run needlefish-weekly-eval。

4.7 Live 驗證(埋 bug PR 流程)

任何 GitHub adapter 的行為改動,單測綠了之後還要 live 目測一次(GitHub UI 的實際長相無法單測):

  1. 在測試 repo(或本 repo)開 branch,故意埋一個明確的 bug(例:改掉一個邊界檢查),開 PR。
  2. 等 review.yml 跑完(或 @needlefish recheck),核對:verdict 正確、行內留言錨對行、check-run 狀態正確。
  3. 驗 sticky:修掉一半 bug 再 push,核對 fresh/open/resolved 標記與 verdict 轉變。
  4. 驗完關 PR、刪 branch。先例:PR #8。

W7 · Harness 煞車系統(ACP 線)

獨立線 · 可與 W2–W5 並行 階段 A:1 天;階段 B:1 天 風險:中(碰 runner 子行程管理)

設計原則(一句話版)

「ACP 是訊號線,煞車系統必須在 Needlefish。」Runner(codex/claude/opencode/未來的 ACP agent)是不可信 agent:可能掛死、可能不理 cancel、可能生孫行程。可靠性邊界由 Needlefish 擁有:deadline → soft cancel → grace → kill process tree,OS 級 kill 是最後保證,協議級 cancel 只是禮貌。完整設計圖:docs/plans/needlefish-acp-harness-design.html(W0 移入後的路徑)。

階段 A — RunnerProcess 硬保證(先做,獨立有價值,不需 ACP)

改 src/shared/runner-process.ts(既有 timeout 邏輯在此):

  1. spawn 時把 runner 放進自己的 process group(detached: true),記 pgid。
  2. timeout 到:先對群組送 SIGTERM(process.kill(-pid, "SIGTERM"))→ 等 grace(寫死 5 秒)→ 仍活著就 SIGKILL 群組。darwin 與 linux 行為一致;不支援 Windows(現況也不支援,不新增)。
  3. 被 kill 的 run 回傳 typed timeout error(沿用既有錯誤型別/訊息慣例,見檔內現有 timeout 路徑),上層照既有失敗路徑走(GitHub mode 貼 failed check;絕不默默當成功)。
  4. 殭屍測試(DONE 的核心判準):stub script spawn 一個孫行程、trap SIGTERM 不退出;斷言 deadline 後(a)孫行程也死了(用 pgid probe),(b)review 收到 typed error 而非 hang,(c)測試本身在有限時間內結束。放 src/shared/runner-process.test.ts,沿用 temp-script stub 慣例。

階段 B — ACP adapter(A 出貨後,立項前跟 Frank 確認一次)

  1. 新增第五種 runner:acp(現有:codex/claude/opencode/openai/grok,見 src/shared/runner.ts)。ACP 提供 cancel/update 的協議語意;deadline 與 kill 仍走階段 A 的機制——協議 cancel 先試,不聽話就 OS kill。
  2. read-only 契約照第 5 條:ACP agent 的沙箱/權限比照 codex runner 的處理(token env 剝除、唯讀)。
  3. 測試:stub ACP server(temp script)模擬四個象限——clean exit / typed error / malformed output / hang,四種都必須以確定性結局收場。
  4. eval:用 --runner acp 跑 smoke 子集確認管線通;不需 prompt gate(不動 prompts)。

R8 · 修復 lane(--fix 的終局形態)— 架構已定案

架構決策(2026-07-04,Frank 拍板):同 repo、分 lane。Needlefish 不自己跑修復 loop——裁判與球員分離,reviewer 當 oracle,fixer 是它的獨立消費者。既不把 --fix 塞進 review 流程(污染 read-only 信任敘事),也不過早開新專案(重複一整套 harness/沙箱/eval 基礎設施)。分 lane 靠以下四條硬規則維持邊界,違反任何一條 = 架構腐蝕,停工回報:

  1. 依賴單向。fixer 可以 import core/shared;reviewer 核心永不 import fixer 的任何東西(檢查法:grep -rn 'from ".*fix' src/core src/shared src/adapters 必須零命中)。fixer 消費 reviewer 一律走 CLI + W8 的 JSON 契約,不走函式呼叫——這保證將來抽離是搬目錄,不是動手術。
  2. 獨立指令、獨立沙箱身分。修復是獨立入口(needlefish fix 子指令或獨立 bin),有自己的權限剖面(可寫入自己開的 branch、可跑目標 repo 測試 = 執行不可信程式碼,需要自己的沙箱設計,威脅模型獨立成文)。action.yml 的 review 路徑永不載入任何 write 能力的程式碼——Marketplace 的「read-only by construction」宣稱不得從「構造上不可能」弱化成「我們保證沒呼叫」。
  3. 裁判獨立性。re-review 對 fix 的審查,調用時不得攜帶「這是我自己改的」上下文;fixer 應使用與 reviewer 不同的模型(打破盲點相關性:同模型血統的 fixer 犯的錯,reviewer 大概率因同一偏誤看不見——go-backend-slop-swallow 是實證。選型依據:RESULTS.md 全模型表)。
  4. 抽出條件寫死。fixer 出現自己的發版節奏,或出現第二個消費者(別的 reviewer/工具想用這個 fix loop)→ 抽成獨立專案。屆時因規則 1,成本是搬目錄 + 把 core/shared 抽成共用套件。

Loop 煞車規格(階段 3 之前就要寫死,W7 的煞車思維升到 loop 層級):

  • 終止條件(全部確定性):圈數硬上限 3;「連續兩圈 findings 集合無縮小」即棄;單 PR token/時間預算上限;任何一項到頂 → 棄圈,留下乾淨狀態(fix branch 刪除或標 draft + 失敗報告),永不留半改的 branch,永不碰原 branch。
  • 誤報成本不對稱:fixer 只消費通過資格判準的 finding——severity ∈ {P0,P1,P2}、有 validation 步驟、cross-file 者必有 consumer 引用。P3 與 residual risk 永不自動修。
  • 第二套 eval 先行:進入階段 3 前必須有 fix eval harness(fixtures 帶可執行測試;量:修復率、測試通過率、新 bug 引入率)。沒有這套 harness,階段 3 沒有 gate 可寫,等於裸奔——不准做。

信任階梯三階段(每階段獨立 gate,上一階段的實測數據是下一階段的入場券):

  1. 階段 1(已含於 W2):suggestion 區塊,人類點擊才 commit。入場數據:採用率 ≥ 30%(低於此 = 建議品質不夠格談自動化,停在這裡繼續改品質)。
  2. 階段 2:fix branch。@needlefish fix 指令(maintainer-only)→ fixer 在新 branch 套用驗證過的 replacement + 跑目標 repo 測試 → 測試綠才開 PR 給人審,不綠附失敗報告。gate(現在寫死):20 次真實 fix PR 中,人類 merge 率 ≥ 60% 且零起「fix 引入新 bug」事件。前置:W8 已出貨(fixer 的輸入就是那個 JSON 契約)。
  3. 階段 3:自動修復 loop。changes_requested 時自動跑 fix → re-review 收斂迴圈(煞車規格如上),產出 fix PR。入場券 = 階段 2 的 gate 數據 + fix eval harness 存在 + Frank 明示同意。在階段 2 數據存在之前,任何人(包括未來的你)提議直接做階段 3,一律拒絕——這是本文件預先行使的否決權。

放棄條件:階段 1 採用率連續兩季 < 15% → 人類根本不想要機器改碼,整條線降級為永久 suggestion-only,FUTURE_TODO 記錄數據。

R12 · Eval 公開 benchmark 化

觸發:fixtures ≥ 60、defect-class 覆蓋表完成、R9 至少跑過一輪(證明 eval 能公平比較模型)。為什麼值得:RESULTS.md 的全模型表已經是市面上少見的「review agent 真實 recall」數據;公開它 = Needlefish 從工具變成該領域的度量衡,這是比任何行銷都強的護城河。步驟骨架:fixtures 抽成獨立可跑的公開子集(私留 holdout 防洩題)+ 排行榜頁(靜態生成,不要建服務)+ 投稿協議(新 runner 一個 PR 就能上榜)。紅線:公開集永不含任何真實 repo 衍生的內容(miss museum fixture 要脫敏重寫);holdout 永不公開。

R13 · 信任量測與「敢擋 merge」

觸發:≥ 5 個活躍 repo 在用。量什麼(全部從 GitHub API 可得,不建服務):override rate(changes_requested 後人類未修就 merge 的比率)、fix-before-merge rate(finding 被修掉才 merge)、suggestion 採用率、needs_human 的人工結論分布。做法:一個 read-only 統計腳本(scripts/)+ 每月人工判讀,寫進 RESULTS.md 姊妹檔 eval/FIELD.md。敢擋 merge 的判準(寫死):override rate 連續一季 < 10% 且零 P0 誤擋投訴 → 才可以在 README 提供「required check 建議配置」;在那之前,Needlefish 永遠自稱 advisory。這條線沒有放棄條件——它就是北極星本身的量測。

R14 · 審查知識庫閉環(最遠的一步)

觸發:miss museum 累積 ≥ 20 個真實 miss fixture。假說:屆時 fixtures 會呈現缺陷類別的長尾分布,而 TRIGGER A–D 只覆蓋頭部。方向:把 miss museum 蒸餾成缺陷類別覆蓋表(哪些類別有結構化協議、哪些裸奔)→ 每季挑「真實 miss 最多的裸奔類別」做一輪 P9 式的 TRIGGER 工程(六輪迭代筆記是方法論教材)。紅線不變:注意力預算——TRIGGER 總篇幅有上限,加新的可能要求砍舊的(屆時用 eval 驗證砍誰);prompt 不是倉庫,是稀缺的注意力分配。

給未來規劃者的收尾規則(本文件的規劃到此為止)

  • H2/H3 全部做完或全部觸發失敗時,這份文件的規劃壽命結束。屆時的規劃方法:拿三個月的真實使用數據(R13 的量測),找「信任階梯上最痛的一階」,只規劃那一階。這就是本文件每一輪規劃實際使用的方法。
  • 永遠不要同時開超過兩條線(一條碰模型品質、一條純工程,是唯一安全的並行組合)。
  • 任何規劃寫下來之前先問:「這一項的 gate 是什麼?失敗長什麼樣?」答不出來的項目不准進 roadmap。

§6 故障手冊(Playbooks)

PB1 · Eval gate 失敗

  1. 最多迭代兩輪(§0 第 5 條)。第二輪仍不過:revert 改動——prompt 改動用 git checkout <基線 SHA> -- prompts/;已 commit 的用 git revert <sha>。
  2. RESULTS.md 記錄(模板,仿 gating-sweep 段): 
    ## <工單名> gate (YYYY-MM-DD) — REVERTED
<一句話:試了什麼、判準、實測數字、為何不過>
Reports: eval/reports/<...>.json
  3. 失敗數據是資產,不是恥辱——gating sweep 的 revert 紀錄後來救過至少一次重複立項。

PB2 · promptHash 不可比陷阱

症狀:兩份報告 recall 差很多,結論卻怪怪的。先查兩份的 promptHash 欄位——不同 hash 的數字沒有可比性(RESULTS.md 頂部的規則)。需要跨版本比較時,在當前 hash 下重跑基線臂。

PB3 · 單抽閃爍

全量單抽差 1–2 個 fixture = 雜訊,不是訊號(medium effort 的天然方差)。對 divergent fixtures --draws 3 confirm 後才准下結論(第 6 條)。

PB4 · Weekly issue:recall dropped

  1. 開兩份報告(issue body 有路徑),diff recallByFixture 找掉的 fixture。
  2. 對掉的 fixture confirm × 3。噪音 → 關 issue 附數據。真回歸 →
  3. 檢查兩份報告的 promptHash 是否相同。不同 → 期間有 prompt/管線 commit,git log --oneline -- prompts/ src/core/ src/shared/ 逐一嫌疑,必要時對嫌疑 commit 前後各跑相關 fixture 子集定位。
  4. 相同 hash 仍回歸 → 上游模型(codex CLI 或模型本身)變了:檢查 runner 版本,考慮 pin runner_version;RESULTS.md 記錄環境變異。

PB5 · Weekly issue:run failed / self-hosted runner 死了

  1. 看 issue 連結的 Actions log:是 eval 程式錯(→ 修碼)還是 runner 機器問題(→ 下一步)。
  2. 機器問題:SSH 上 runner 主機檢查 GitHub Actions runner service、磁碟、codex CLI 登入態(~/.codex/auth.json)。重佈:scripts/deploy-ubuntu.sh。
  3. 機器短期救不回:weekly 可暫改 runs-on: ubuntu-latest + CODEX_AUTH_JSON secret 走 hosted(較慢但能動),修好再改回。

PB6 · 發錯版

git tag -f <浮動tag> <上一個好版 SHA> && git push -f origin <浮動tag>。不可變 tag 不回收;錯得離譜的 release 用 gh release edit 標 prerelease + 說明。

PB7 · GitHub 貼文失敗

行內留言整批失敗 → body-only fallback 是內建行為(v1 設計),check-run 仍會貼。若連 check-run 都失敗:查 workflow 的 permissions(需要 checks: write、pull-requests: write)與 token 是否過期。降級永遠優於沉默——任何改動不得讓「審了但什麼都沒貼」發生。

PB8 · Codex CLI 升級弄壞東西

action 有 runner_version input 可 pin 舊版止血。診斷期間可用備援 runner 對照(RESULTS.md 全模型表:次佳是 claude opus-4.7 @xhigh,recall 76%;opencode 系列的高 invalidJson 是 runner 迴圈問題,不是模型品質——RESULTS.md Notes 有完整說明,別誤判)。

PB9 · 「模型看得到證據卻漏抓」型 miss

先讀 RESULTS.md 的 P9 六輪迭代筆記——大概率你要處理的形狀已經出現過(critic 誤刪 → 窄例外;零呼叫端逃逸 → public-carrier 條款;trigger-as-taxonomy → 反分類學句;措辭盪過頭 → 收窄範圍)。然後才設計新手段,且照第 12 條:結構優先於措辭。

§7 何時必須停下來問 Frank(升級規則)

  • 一律要問:push / merge / deploy / 刪除任何檔案或 branch / 碰 secrets / 對外發布(Marketplace、release、對外文案)/ 新增依賴 / 改安全模型(沙箱、token 處理、權限)/ 改 deriveVerdict 語意。
  • 發現本文件與現實衝突:以現實為準,停下報告,不要照文件硬做。
  • 工單之外的「好主意」:寫下來放 FUTURE_TODO,不執行。
  • 連續兩次 gate 失敗後想試第三次:停。那個衝動就是第 1 條要防的東西。
  • 不需要問的(標準授權):讀任何東西、跑測試/eval、本地 commit、read-only 調查、按本文件工單順序執行已寫死的步驟。

§8 明確不做的事(附理由,防止後人重新發明)

  • 跳級的全自動 --fix — 修復 lane 的架構與階梯已定案(R8:同 repo 分 lane、三階段各有 gate);「不做」的是跳過階段數據直接上自動 loop,以及任何把 write 能力混進 reviewer/review action 的做法。
  • Bot 給 APPROVE — pass 永遠是 COMMENT + success check;bot 自我批准是治理反模式。
  • Schema 驗證依賴(Zod 等) — normalize.ts 小而可讀;依賴是永久維護成本。
  • Plugin 框架 / 廣義抽象 — 現有 adapters/core/shared 分層足夠;第二個真實需求出現前不抽象。
  • Token 精算儀表板 / 趨勢 UI — weekly JSON + RESULTS.md 敘事已滿足決策;儀表板是給沒有紀律的團隊的安慰劑。
  • 對 fixture 打補丁式的 prompt 修改 — 憲法第 2、3 條的存在理由。
  • 增量 diff 審查(只審新 commit) — 會漏跨 commit 交互;全量 48s 夠便宜,省的該是噪音(sticky review 已解決)不是 token。
